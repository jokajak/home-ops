## Goal

Two outcomes:

1. **Grafana uses Authentik for SSO** (OIDC generic OAuth).
2. **Authentik itself can be logged into with a social identity** — add a social-login
   *source* on the Authentik login screen, linked to the existing Authentik account by
   email. **Current focus: GitHub** (Josh has a GitHub OAuth app; Google is deferred —
   no Google OIDC client exists yet).

## Architecture notes (how Authentik social sources work)

- A `authentik_source_oauth` with `provider_type = "github"` adds a "Login with GitHub"
  button. With the built-in `github` provider type, Authentik already knows GitHub's
  authorize/token/profile URLs — **do not** set `oidc_jwks_url` (the old config pointed it
  at `token.actions.githubusercontent.com`, which is GitHub *Actions* CI-token OIDC, not
  user login). GitHub calls back to
  `https://auth.${SECRET_DOMAIN}/source/oauth/callback/github/` (slug = `github`).
- For the button to appear, the source must be listed in the **identification stage**'s
  `sources` (`authentik_stage_identification.authentication-identification`, `stages.tf`).
  It previously set `show_source_labels = true` but did **not** set `sources`, so no social
  button rendered.
- `authentication_flow` for a source should be the built-in `default-source-authentication`
  flow (not the provider-authorization flow the old config reused).
- To log into the *existing* account (not create a new user), set
  `user_matching_mode = "email_link"` so a GitHub login with a matching email links to the
  existing Authentik user.
- `enrollment_flow` governs a brand-new identity; we reuse `authentik_flow.enrollment-invitation`
  (invitation-gated), so there's no open enrollment.

### Root fix — provider/server version skew (the real blocker)

Renovate had bumped the **`goauthentik/authentik` provider to `2026.5.0`**, but the running
**Authentik server is `2024.12.x`** (Helm chart `2024.12.0` in
`kubernetes/apps/security/authentik`). An ~18-month-newer provider speaks a newer API
schema than the server, which surfaced as two distinct failures:

- `tofu validate` errored on `authentik_group.parent` (the 2026 provider renamed it to a
  list `parents`).
- Once auth worked, every authentik **data-source read** failed with
  `HTTP Error 'no value given for required property autocomplete'` (the 2024.12 server's
  API responses lack fields the 2026 provider's client marks as required).

Initially (wrongly) patched the first symptom by switching the four groups to
`parents = [...]`. **Correct fix:** pin the provider back to the server line —
`version = "2024.12.0"` in `versions.tf` — and revert the groups to scalar `parent`
(`grafana_editors`, `grafana_viewers`, `monitoring` in `application_grafana.tf`; `media`
in `directory.tf`). Re-locked for darwin/linux amd64+arm64. `tofu validate` passes.

> Keep the provider in lockstep with the Authentik chart; do not let renovate bump it
> ahead of the server. (Upgrading the *server* 2024.12 → 2026.x is a separate, riskier
> effort — many releases of migrations on a live SSO — and is out of scope here.)

## Gotcha — Bitwarden provider auth (resolved)

After a long gap, `tofu plan` failed every Bitwarden read with `Vault is locked`,
even though `bw login --apikey` + `bw unlock` worked **by hand** with the same SOPS
creds (BW_PASSWORD/CLIENTID/CLIENTSECRET/EMAIL all confirmed current). Root cause: the
`maxlaverse/bitwarden` provider (bumped 0.17.2→0.17.6) shelling out to the official `bw`
CLI couldn't reuse the session from its own unlock — a CLI⇄provider version mismatch.
Also note: a manually-set `BW_SESSION` conflicts with `master_password` (`session_key`
is a separate, mutually-exclusive auth path), and the provider only self-unlocks from a
**logged-out** CLI state. **Fix:** switched the provider to the embedded client
(`client_implementation = "embedded"` in `provider.tf`) — talks to the Bitwarden API
directly, no `bw` CLI, same SOPS creds. No `BW_SESSION` should be set when running.
