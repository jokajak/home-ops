## Goal

Two outcomes:

1. **Grafana uses Authentik for SSO** (OIDC generic OAuth).
2. **Authentik itself can be logged into with a Google identity** ‚Äî i.e. add Google as a
   social-login *source* on the Authentik login screen, linked to the existing Authentik
   account by email.

## TL;DR of current state (verified 2026-06-14)

- **Goal 1 (Grafana SSO) is essentially already built in-repo** ‚Äî both sides exist:
  - Authentik side: `terraform/authentik/application_grafana.tf` defines the
    `authentik_provider_oauth2.grafana_oauth` provider, the `authentik_application`,
    the groups (`Grafana Admins/Editors/Viewers`, `Monitoring`), and policy bindings.
    The OIDC client_id/secret are generated by the `oidc_creds` module and stored in
    Bitwarden item **`authentik-client-grafana`**.
  - Grafana side: `kubernetes/apps/observability/grafana/app/helmrelease.yaml`
    `grafana.ini.auth.generic_oauth` is fully configured (auth/token/userinfo URLs,
    PKCE, group‚Üírole mapping). `externalSecret.yaml` pulls `authentik-client-grafana`
    (`username`‚Üí`GF_AUTH_GENERIC_OAUTH_CLIENT_ID`, `password`‚Üí`..._SECRET`).
  - ‚áí Remaining work for Goal 1 is **verification only** (confirm it actually works on
    the live cluster), not new code.
- **Goal 2 (Google login to Authentik) is NOT built.** `terraform/authentik/main.tf`
  defines a `github` source only. There is a placeholder
  `authentik_policy_expression.google_username` (currently `return False`) but **no
  `authentik_source_oauth` for Google**, and no Google source is bound to the
  identification stage.

So the bulk of the actual new work is **Goal 2**.

## Architecture notes (how Authentik social sources work)

- A `authentik_source_oauth` with `provider_type = "google"` adds a "Login with Google"
  button. Google calls back to
  `https://auth.${SECRET_DOMAIN}/source/oauth/callback/google/` (slug = `google`).
- To make the button appear, the Google source must be listed in the **identification
  stage**'s `sources` (`authentik_stage_identification.authentication-identification` in
  `stages.tf` ‚Äî it currently sets `show_source_labels = true` but does **not** set
  `sources`, so social buttons may not render today).
- To log into the *existing* account (not create a new user), set the source's
  `user_matching_mode = "email_link"` so a Google login with a matching email links to the
  existing Authentik user.
- `enrollment_flow` governs what happens for a brand-new identity. For a single-admin
  homelab we want existing-user linking; enrollment can reuse the existing
  `authentik_flow.enrollment-invitation` (as github does) or be left to link-only.

## Implementation plan (Terraform ‚Äî `terraform/authentik/`)

- [ ] **Add variable** `google_oauth_credentials_id` to `variables.tf`.
- [ ] **Add Google source** in `main.tf` (or a new `source_google.tf`):
  - `data "bitwarden_item_login" "google_oidc_creds"` keyed on
    `var.google_oauth_credentials_id`.
  - `resource "authentik_source_oauth" "google"` ‚Äî `provider_type = "google"`,
    `slug = "google"`, `consumer_key`/`consumer_secret` from the Bitwarden item,
    `authentication_flow = data.authentik_flow.default-authorization-flow.id`,
    `enrollment_flow = authentik_flow.enrollment-invitation.uuid`,
    `user_matching_mode = "email_link"`.
- [ ] **Bind source to the login screen**: set `sources` on
  `authentik_stage_identification.authentication-identification` (`stages.tf`) to include
  the built-in source **and** `authentik_source_oauth.google.uuid`.
  (Confirm whether the existing `github` source also needs adding here ‚Äî it currently is
  not listed, so the github button likely doesn't render either. Decide whether to keep
  github.)

  `authentik_policy_expression.google_username` (`return False`) with a real mapping per

  `email_link` matching makes it unnecessary.
